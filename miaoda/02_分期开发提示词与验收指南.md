# 秒哒分期开发提示词与验收指南

这份文档给你自己使用，用来分阶段要求秒哒开发。核心原则只有一个：

```text
秒哒输出所有文本和字幕包，GitHub Actions 不跑 ASR。
```

## 总体路线

分四期：

1. 第一期：单条媒体解析/转写 API。
2. 第二期：文件下载、缓存、幂等和可校验 bundle。
3. 第三期：每日采集 `/v1/daily/collect`，直接输出日报所需全部文本。
4. 第四期：GitHub Actions 只下载秒哒产物，校验后生成日报并写飞书。

只有第四期通过真实 workflow run，才算 GitHub Actions 端到端跑通。

## 第一期：单条媒体转写

目标：输入一个小宇宙或 YouTube URL，输出真实逐字稿和时间戳。

必须支持：

- `POST /v1/media/extract`
- `GET /v1/media/extract/{job_id}`
- 小宇宙 episode URL
- YouTube video URL
- `transcript_text`
- `segments`
- `text_chars`
- `sha256`
- `duration_seconds`
- `last_timestamp_seconds`
- `coverage_ratio`

### 第一期给秒哒的提示词

```text
请开发“媒体解析与转写服务”的第一期版本。严格按《01_媒体解析转写服务需求文档.md》实现，不要做日报总结、飞书写入、复杂前端或多用户系统。

第一期只做单条媒体：
1. POST /v1/media/extract：创建异步解析/转写任务。
2. GET /v1/media/extract/{job_id}：查询任务状态和结果。
3. 小宇宙 episode：解析页面内嵌数据，找到真实 audio_url，由秒哒服务器侧下载/读取音频并 ASR。
4. YouTube video：优先官方字幕/自动字幕；没有字幕时由秒哒服务器侧 ASR。
5. 返回字段：status、platform、source_url、audio_url、title、source_name、published_at、duration_seconds、language、transcript_text、segments、text_chars、sha256、last_timestamp_seconds、coverage_ratio、source_method、completed_at。
6. sha256 必须按 transcript.txt 的 UTF-8 bytes 计算。
7. success 必须要求 transcript_text 非空、segments 非空、coverage_ratio >= 0.95。

第一期验收只看这 3 个小宇宙链接：
https://www.xiaoyuzhoufm.com/episode/6a41f1212e335a35a80b0159
https://www.xiaoyuzhoufm.com/episode/6a40b830ffcf45052c805d5f
https://www.xiaoyuzhoufm.com/episode/6a412def9d2f5743683f320e

注意：逐字稿必须来自官方字幕、平台自动字幕或 ASR，不能用大模型根据简介/时间线/标题补写。
```

### 第一期验收

向秒哒要：

- 服务地址
- API token
- 3 条小宇宙的 `job_id`
- 3 条小宇宙的 `transcript.txt`
- 3 条小宇宙的 `transcript.srt` 或 `transcript.vtt`
- 3 条小宇宙的 `sha256`
- 3 条小宇宙的 `coverage_ratio`

不合格：

- 只有摘要，没有完整逐字稿
- 没有时间戳
- 没有 `sha256`
- 没有 `coverage_ratio`
- 小宇宙只解析简介时间线，没有转写音频

## 第二期：文件下载和缓存

目标：单条媒体结果可下载、可复用、可校验。

必须新增：

- `GET /v1/media/extract/{job_id}/files/transcript.txt`
- `GET /v1/media/extract/{job_id}/files/transcript.srt` 或 `.vtt`
- `GET /v1/media/extract/{job_id}/files/metadata.json`
- `GET /v1/media/extract/{job_id}/files/extraction.json`
- `GET /v1/media/extract/{job_id}/files/bundle.zip`
- `GET /v1/media/extract/by-url?url=...`
- `Authorization: Bearer <token>`
- 幂等缓存

### 第二期给秒哒的提示词

```text
请在第一期基础上实现第二期：文件下载、幂等缓存和可校验 bundle。

新增接口：
1. GET /v1/media/extract/{job_id}/files/transcript.txt
2. GET /v1/media/extract/{job_id}/files/transcript.srt 或 transcript.vtt
3. GET /v1/media/extract/{job_id}/files/metadata.json
4. GET /v1/media/extract/{job_id}/files/extraction.json
5. GET /v1/media/extract/{job_id}/files/bundle.zip
6. GET /v1/media/extract/by-url?url=<url>

新增要求：
1. 实现 Authorization: Bearer <token>。
2. 同一个 normalized_url + language + need_transcript 重复提交时，直接返回已有成功 job，不重复 ASR。
3. bundle.zip 必须包含 metadata.json、extraction.json、transcript.txt、transcript.srt 或 transcript.vtt。
4. extraction.json 必须包含 status、method、language、attempts、duration_seconds、last_timestamp_seconds、coverage_ratio、text_chars、sha256、completed_at、error_type、error_message。
5. 下载 transcript.txt 后，本地复算 sha256 必须一致。
```

### 第二期验收

重点检查：

- 重复提交是否命中缓存
- `bundle.zip` 是否完整
- `sha256` 是否可复算
- 失败时是否有 `error_type` 和 `error_message`

## 第三期：每日采集

目标：秒哒直接输出某一天日报所需的所有数据和所有文本。

这是最终 GitHub Actions 自动化的主接口。

必须实现：

- `POST /v1/daily/collect`
- `GET /v1/daily/collect/{job_id}`
- `GET /v1/daily/collect/{job_id}/files/daily_items.json`
- `GET /v1/daily/collect/{job_id}/files/manifest.json`
- `GET /v1/daily/collect/{job_id}/files/subtitles_bundle.zip`

时间窗口：

```text
日报日期 D = 北京时间 D-1 06:00:00 到 D 06:00:00
```

### 第三期给秒哒的提示词

```text
请在第二期基础上实现第三期：每日采集接口 /v1/daily/collect。

这个接口是 GitHub Actions 的主路径。GitHub Actions 不跑 ASR、不调用 yt-dlp、不下载音频，只调用秒哒并下载结果。

新增接口：
POST /v1/daily/collect
GET /v1/daily/collect/{job_id}
GET /v1/daily/collect/{job_id}/files/daily_items.json
GET /v1/daily/collect/{job_id}/files/manifest.json
GET /v1/daily/collect/{job_id}/files/subtitles_bundle.zip

请求支持：
- date
- timezone
- window_start
- window_end
- sources_profile
- require_transcripts
- allow_asr

输出要求：
1. daily_items.json 字段兼容 podcast-distill：platform、category、source_name、source_url、title、original_title、url、published_at、duration、description、chapters。
2. subtitles_bundle.zip 必须可直接解压成 subtitles/ 目录。
3. subtitles/ 中每个 5 分钟以上条目必须有 .json、.txt、.srt 或 .vtt。
4. 每个 subtitles/*.json 必须有 url、text、text_chars、sha256、duration_seconds、last_timestamp_seconds、coverage_ratio。
5. manifest.json 必须说明 source_count、item_count、success_count、failure_count、window_start、window_end、errors。
6. require_transcripts=true 时，任何必需条目没有完整字幕/转写，都不能返回 success。

验收日期：2026-06-30。
窗口：2026-06-29 06:00:00+08:00 到 2026-06-30 06:00:00+08:00。
必须包含并转写成功这 3 条小宇宙：
https://www.xiaoyuzhoufm.com/episode/6a41f1212e335a35a80b0159
https://www.xiaoyuzhoufm.com/episode/6a40b830ffcf45052c805d5f
https://www.xiaoyuzhoufm.com/episode/6a412def9d2f5743683f320e
```

### 第三期验收

重点检查：

- 时间窗口是否严格
- 是否混入其他日期
- `daily_items.json` 是否完整
- `subtitles_bundle.zip` 是否能直接解压给日报脚本用
- 小宇宙是否真实 ASR，不是简介
- YouTube 是否优先字幕，没有字幕才 ASR
- `manifest.json` 是否列出失败原因

## 第四期：GitHub Actions 跑通

目标：GitHub Actions 只下载秒哒产物，然后生成日报并写飞书。

不要求 GitHub Actions 跑：

- ASR
- whisper
- `extract_subtitles.py`
- `yt-dlp`
- 音频下载

### 第四期给秒哒/后续模型的提示词

```text
请基于秒哒 /v1/daily/collect，给出 GitHub Actions 端到端跑通方案。

硬性要求：
1. GitHub Actions 只调用 /v1/daily/collect。
2. GitHub Actions 下载 daily_items.json、manifest.json、subtitles_bundle.zip。
3. GitHub Actions 解压 subtitles_bundle.zip 到 subtitles/。
4. GitHub Actions 校验 subtitles/*.json 中的 text_chars、sha256、coverage_ratio。
5. 校验通过后才运行 scripts/generate_daily_report.py --require-transcripts。
6. 生成成功后才运行 scripts/publish_feishu.py。
7. 任一必需条目缺字幕、sha256 不一致、coverage_ratio < 0.95 时，直接失败，不发布飞书。
8. 无论成功失败都上传 artifacts。

不允许在 GitHub Actions 中安装 whisper、跑 ASR、下载音频或调用 extract_subtitles.py。

验收时必须提供一次手动触发 workflow_dispatch 的真实 run 结果，并检查 artifacts：
- reports/daily_items_日期.json
- reports/daily_items_日期.manifest.json
- subtitles/
- reports/daily-日期.json
- reports/daily-日期.md
- 飞书发布日志
```

## 最终自动化形态

最终链路应该是：

```text
GitHub Actions 定时触发
  -> 调用秒哒 /v1/daily/collect
  -> 轮询 job
  -> 下载 daily_items.json / manifest.json / subtitles_bundle.zip
  -> 解压 subtitles_bundle.zip 到 subtitles/
  -> 校验 text_chars / sha256 / coverage_ratio
  -> generate_daily_report.py --require-transcripts
  -> publish_feishu.py
  -> 上传 artifacts
```

GitHub Actions 不应该承担：

- 大量 YouTube 抓取
- 小宇宙长音频 ASR
- 本地 whisper 推理
- 重复下载同一音频
- 保存浏览器 cookie

现有 `.github/workflows/daily-digest.yml` 是旧方案参考，不是最终验收方案。

## 给秒哒的总提醒

- 这不是摘要工具，是解析与转写服务。
- 逐字稿必须真实。
- 秒哒必须输出所有文本。
- 结果必须可下载、可校验、可缓存。
- 失败必须明确原因。
- 不要为了返回 success 编造内容。
